# Execute a Test Patch in an isolated sandbox before review, gating on a separate attempt budget

## Status

accepted

Introduces the **Patch Execution** and **Execution Attempts** glossary terms in `CONTEXT.md`, and reconciles the now-stale "_Avoid_: Test execution" lines on **Patch Review** and **Code Reviewer** (which previously defined execution away). Extends — does not reverse — the score/escalation loop of ADR [0004](0004-scored-review-escalates-to-human.md): exhaustion still escalates to the owner and never fails the run. Reads the checkout under the single-writer claim of ADR [0007](0007-single-writer-checkout-claim-per-repository.md) and adds one node to the unified graph of ADR [0002](0002-intent-routed-unified-langgraph.md). Adds the `executing` Stage marker to the Agent Stream vocabulary.

## Context and decision

Patch Review is static: the Code Reviewer scores a Test Patch against retrieved Repository Documents but never runs the tests, so a patch that imports a missing symbol, mis-uses a fixture, or simply fails its own assertions can still escalate to the owner looking plausible. We want the owner to review tests we have actually *seen pass*. The generated test code is untrusted — repository content can carry prompt injection, and the model can emit anything — and the backend process holds every secret it owns in its environment (`POSTGRES_PASSWORD`, `REPOSITORY_TOKEN_ENCRYPTION_KEY`, the decrypted `QA_GIT_TOKEN`, LLM keys) plus write access to every user's checkout. Running that code anywhere near the backend process is the whole risk.

We decided to add a **Patch Execution** gate between generation and review, executed in a hardened, disposable Docker sandbox, on its own attempt budget:

- **Dual-Zone separation.** The host keeps a persistent checkout used only for cloning, indexing, and deriving the Test Patch — code is never executed there. Execution happens only inside an ephemeral sandbox container that receives a **read-only mount of the checkout** and copies it into its own writable layer. The real checkout is never writable to untrusted code, and is restored to the indexed commit regardless of outcome.

- **Gate, between generation and review, on a separate budget.** A new `execute_tests` node sits on the `generate_code → review_patch` edge. Tests pass → on to Patch Review, whose post-review logic is unchanged. Tests fail → back to `generate_code` carrying the captured failure output, up to **`MAX_EXECUTION_ATTEMPTS` (default 4)** — a budget kept **distinct from the Generation Retries** (default 2) that the review-score loop spends. Only a run whose tests *actually executed and failed* spends an Execution Attempt. Because `generate_code` distinguishes its first pass from a revision by the presence of `review_result`, an execution-driven retry (which loops back *before* review runs) must be detected separately via `test_failure_output`/`execution_attempts`, so it skips branch preparation and calls `code_generator.revise(...)` with the test output rather than review findings.

- **Two-phase, because install needs the network and the untrusted run must not have it.** Phase 1 provisions with the network enabled but **executes no test code** (`uv` installs the repo's Python version and dependencies). Phase 2 runs `pytest` with `network_mode="none"`. Network access cannot be toggled mid-container, so the phases are two containers sharing one workspace volume. `--network none` in phase 2 is load-bearing: it denies untrusted code the Docker network (where `db` and `weaviate` are reachable container-to-container), the cloud instance-metadata endpoint (`169.254.169.254`, i.e. the host's IAM credentials), and any call-home/exfiltration path — none of which a scrubbed environment alone prevents.

- **`uv` is the single universal provisioner.** It installs the repo's exact Python version (`.python-version` / `pyproject` `requires-python`) and dependencies, preferring the repo's own lockfile (`uv.lock → uv sync`), else `pyproject`/`requirements.txt → uv pip install`, with a pip fallback. One cross-platform binary covers uv/pip/poetry/pdm/pipenv repos with minimal branching; we do **not** install poetry/pipenv inside the sandbox.

- **Run only the generated Test Files; pass is exit 0 with collection.** The gate runs exactly the patch's Test Files, so the Repository's pre-existing or flaky suite never blocks the patch. Pass = exit 0, at least one test collected, zero failures; a collection/import error or any failure is a phase-2 failure that feeds revision.

- **Distinguish "tests failed" from "could not run", and never fail the run.** A **phase-2** failure (tests ran, failed/errored) revises against the Execution Attempts budget. A **phase-1** failure (Docker absent, image/`uv`/network error, setup timeout) or a **phase-2 timeout** takes a **degrade** path: the patch proceeds to review flagged "not executed"/"inconclusive", spends no Execution Attempt, and does not fail the run. Exhausting the 4 attempts likewise proceeds to review flagged "tests failing". The owner always gets an inspectable patch, consistent with ADR 0004.

- **Hardening and disposability.** Phase 2 runs with `cap_drop=["ALL"]`, `no-new-privileges`, `mem_limit`, `nano_cpus`, `pids_limit`, `network_mode="none"`, no `environment=`/`env_file` (so backend secrets never enter the container), and `remove=True` so the writable layer is reclaimed on exit. Phases are bounded by wall-clock timeouts — **phase 1 = 300s, phase 2 = 120s** — enforced by running detached and killing on `wait` timeout. For simplicity the container runs as root *inside* its namespace (still isolated from the host) rather than wiring a non-root uv cache.

- **Setup cost is amortized.** Dependencies are provisioned **once per Coding Run** and the prepared workspace is reused across all Execution Attempts (the generator only rewrites Test Files, never dependencies), so the install runs once, not four times. A persistent host-side `uv` cache is mounted **into phase 1 only** (never the untrusted run phase), so interpreters and wheels are reused across runs and repos.

- **Daemon availability is checked, not assumed.** A startup probe in the FastAPI lifespan logs a clear warning (and surfaces on the health check) when the Docker daemon is absent, but never crashes the app. Each run re-checks and falls to the degrade path if the daemon is down at that moment.

- **Seam and surfacing.** Execution lives behind a `TestSandbox` Protocol with a `DockerTestSandbox` implementation, injected into `build_graph` via a `sandbox_factory` exactly like `workspace_factory`/`publisher_factory`; unit tests inject a fake sandbox, so the suite needs no Docker daemon. The node returns a **structured** result (`{status, exit_code, phase, log_tail}`) — a tail of ANSI-stripped `-q --tb=short` output, ~12 KB, treated as untrusted data when fed to the generator. Execution status (passed / failed / not-executed, counts, log excerpt) is folded into the escalated `ReviewResult` and the `await_decision` interrupt payload, since the after-review path is unchanged.

## Considered options

- **Hard gate that feeds revision vs. advisory signal vs. gate without auto-fix** — chose the hard gate. An advisory signal lets failing tests reach the owner unchanged; a gate that never auto-fixes wastes the model's ability to repair its own output. Coupling execution into the revision loop (with its own budget) means the owner usually sees tests that pass, and only sees failing tests when the model genuinely could not fix them.
- **Between generation and review vs. as part of review vs. after review** — chose between. Executing inside Patch Review overloads a deliberately static concept; executing after review would escalate patches we already know fail. Gating before review keeps Patch Review static and means a patch reaching the owner has cleared both bars.
- **Two containers (two-phase) vs. one container with network** — chose two-phase. A single container cannot both install dependencies (needs network) and deny the untrusted run a network. One networked container is simpler but hands generated code the Docker network, the metadata endpoint, and the open internet; the second container is a cheap price for removing that entire class of attack.
- **Docker vs. nsjail/bubblewrap vs. a remote runner vs. a scrubbed subprocess** — chose Docker. A scrubbed subprocess shares the host kernel, network, and disk — insufficient. nsjail/bwrap are Linux-only and need user-namespace/cap wiring; a dedicated runner service is real infrastructure. Docker Desktop/Engine gives identical, strong isolation on mac/windows/linux from one hardened invocation, and the app already requires Docker.
- **`uv` for everything vs. honoring each native tool** — chose uv-first. Dispatching to poetry/pdm/pipenv per lockfile is the most faithful to each tool's resolver, but bakes more tools into the image and more branch logic; uv-first trades exact per-tool lock fidelity (e.g. `uv sync` will not read `poetry.lock`) for one cross-platform binary, an accepted trade for this demo.
- **Only the generated files vs. the whole suite** — chose only the generated files. Running the whole suite lets a pre-existing or flaky failure block an unrelated patch and is far slower inside the attempt loop.
- **Ephemeral disk overlay vs. `tmpfs` workspace** — chose the disk overlay (`remove=True`). A RAM-backed work dir risks OOM on large codebases; the container's writable layer self-purges on exit without competing for the memory limit.
- **Phase-2 timeout → degrade vs. → revise** — chose degrade. A timeout cannot be cleanly attributed to the generated test versus the repo's own code, so spending an Execution Attempt (and forcing a revision the model may not be able to make) would be unfair; the owner is shown an inconclusive flag instead.
- **Separate Execution Attempts vs. one shared budget with Generation Retries** — chose separate. A test failure and a low review score are different problems; sharing one counter conflates them and makes neither tunable.

## Consequences

- **A Docker daemon is required for the gate to run.** When it is absent (or down at run time) the feature degrades to "not executed"; the rest of the Coding Run is unaffected. The unit test suite needs no daemon (fake sandbox via the seam).
- **Phase 1 still executes lower-tier untrusted code with the network.** Installing the repo's declared dependencies runs their build hooks (`setup.py`, build backends) with network access. This is a different, lower trust tier than the model's generated tests (it is the repo owner's own declared supply chain), but it is a real residual; it runs with dropped capabilities, no backend secrets, and against a copy — not the real checkout. Restricting phase 1's egress to a package index is a future hardening, not done here.
- **Disk-fill is bounded only by the run timeout for now.** `mem_limit` does not cap disk, and a portable per-container disk quota (`storage_opt size=`) is storage-driver-specific, so a pathological test can write to the writable layer up to the time limit. Accepted residual for local/demo use; flagged for a later `storage_opt` guard on supported drivers.
- **A glibc `slim` base is used, not alpine.** musl forces source builds of packages that ship manylinux wheels — slower, flakier, and needing a build toolchain with network. Image size is the only cost of `slim`, and it is worth it.
- **New configuration knobs:** `PATCH_EXECUTION_ENABLED`, `MAX_EXECUTION_ATTEMPTS`, phase-1/phase-2 timeouts, memory/CPU/pids limits, and the runner image reference.
- **One invariant for review and tests:** the untrusted **phase 2** container must always run with `network_mode="none"`, no `environment=`/`env_file`, and against the disposable copy — never the real checkout, never the host network, never the backend's secrets.
