# Question Shape sub-routing on the repository-question branch

## Status

accepted

Extends the unified graph (ADR [0002](0002-intent-routed-unified-langgraph.md)). ADR 0002 routed `classify` to a single-query `retrieve → generate` for every repository question. This ADR inserts a second classifier — `analyzing` — *after* a question has been routed to the `repository_question` branch, replaces the two original nodes with three self-contained retrieval-strategy nodes, and amends the closed Agent Stream Stage vocabulary. It introduces one glossary term, **Question Shape** (`CONTEXT.md`), a sibling of Request Intent. The repository-question branch stays read-only and never reaches the web (unchanged from 0002).

## Context and decision

Naive single-query retrieval misses when the question's vocabulary diverges from the corpus, and it answers multi-part or multi-hop questions poorly because one retrieval cannot cover several distinct information needs. We decided the repository-question branch should first infer the **Question Shape** of the question and then apply a matching retrieval strategy, rather than running one retrieval for everything.

`classify` → (repository_question) → `analyzing` routes to one of three self-contained nodes:

- **`simple`** → `simple_rag`: multi-query + RAG-fusion. Generate N reformulations, run raw hybrid search per variant, merge with Reciprocal Rank Fusion (RRF), then **Cohere-rerank the fused pool against the original question** and hydrate parents.
- **`independent`** → `decompose_parallel`: decompose into independent sub-questions, retrieve per sub-question (single-query hybrid+rerank), answer them in one `llm.batch(...)`, then synthesize.
- **`chained`** → `decompose_recursive`: decompose into dependent sub-questions answered sequentially (IRCoT), each answer fed as context to the next, then synthesize.

`analyzing` is a pure classifier (returns only the shape, `with_structured_output` + cross-provider fallback per ADR 0010, reading `messages` for follow-up context). Uncertain classification falls back to `simple`, which is read-only and side-effect-free — mirroring how 0002's Request Intent falls back to the repository-question branch. Each node owns its retrieval *and* generation; the original `retrieve`/`generate` nodes are deleted and their token-streaming + citation-dedup logic is extracted into a shared helper the three nodes call for their final, streamed answer. Only the final answer (the `simple` answer, or the decomposition synthesis) streams as `Token`s; sub-answers are computed internally. The Stage vocabulary gains `analyzing`, `decomposing`, and `synthesizing`; `retrieving` and `generating` are reused.

## Considered options

- **RRF *feeding* the cross-encoder vs. either one alone.** This is the non-obvious decision. The query-transformation notebooks use RRF precisely because plain Chroma has no reranker; our `DocumentRetriever` already ends with a Cohere cross-encoder. We considered (a) **union → rerank**, dropping RRF — cheapest and high-precision, but it discards the cross-reformulation recall signal RRF captures (a chunk surfacing under several reformulations is evidence of relevance); and (b) **fuse already-reranked parent lists**, calling the existing `retrieve_documents` per variant — minimal new code but N rerank calls and RRF running *after* precision ranking, which is the wrong order. We chose (c): **RRF widens recall across reformulations, then the Cohere reranker makes the final precision call** over the fused pool against the original question, with a single rerank call. RRF and the cross-encoder are both ranking mechanisms, so a future reader will reasonably ask why both exist; they play distinct roles (recall vs. precision) and run in that order. Cost: one new retriever method, `fusion_retrieve_documents`, composed from the existing `search_with_scores` (raw hybrid) and the rerank/hydrate tail.
- **Three self-contained nodes vs. enhancing the existing retrieve/generate pair.** We chose self-contained nodes (including `simple`) for a uniform branch shape, accepting that streaming/citation logic moves into a shared helper rather than living in a single `generate` node. Decomposition interleaves retrieval, per-sub-question answering, and synthesis, so it does not fit the clean `retrieve → generate` split; making all three uniform avoids a hybrid topology where one branch reuses old nodes and two do not.
- **Pure classifier vs. classifier that also emits sub-questions.** We kept `analyzing` a pure classifier and let each decomposition node run its own `decompose` prompt. A combined call would save one LLM call on the decomposition paths but make `simple` pay for a field it ignores, overload the classifier prompt, and leave `decomposing` without a node to emit its stage from. The extra call is negligible next to the per-sub-question retrievals.
- **Single-query vs. fusion retrieval per sub-question.** Sub-questions use single-query hybrid+rerank, not full multi-query+fusion. Decomposition already breaks the ask into simple parts; running fusion per sub-question multiplies LLM and rerank calls combinatorially (N×M) on a read-only, interactive path for little gain.
- **Streaming sub-answers vs. only the synthesis.** Only the final answer streams. `independent` answers its sub-questions with `llm.batch`, which cannot stream token-by-token, and sub-answers are intermediate; streaming them would need new event types beyond `Token`/`Result` and muddy the single-answer contract. Sub-question progress is conveyed by the `decomposing` and `synthesizing` Stage markers instead.
